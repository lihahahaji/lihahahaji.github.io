# 第三章 进程

进程： 可以从三个方面区分： 1.程序、2.数据、3.状态

# 程序的状态：

1.新建（new)

进程在创建时需要申请一个空白的PCB，向其中前些控制和管理进程的信息，完成资源分配。 PCB：进程的“档案”。 如果创建的工作无法被完成，就无法被调度运行，把此时进程所处的状态称为创建状态。

2.运行 3.waiting 等待状态

当进程正在等待合作进程的一个消息时称为等待状态

4.ready 就绪 5.终止

# 进程与线程：

进程-》教学楼 切换代价高 资源分配的最小单元 线程-》教室 切换代价小 程序调度的最小单位

# PCB Process Control BLock 进程控制块：

放在内核空间里 可以看作是进程的代表

记录进程的状态 程序计数器 进程数 CPU寄存器 程序数 内存限制 ... 上下文切换 （context Switch） 上下文 就是 PCB 上下文切换的时间是额外开销（读写内存） -》时间越短越好

# 进程调度

本质上是调度PCB

job queue 工作队列 Ready queue 就绪队列-- 处于就绪状态的进程 Device queue 设备队列-- 等待使用I/O设备的进程

# 短期调度

例如 将设备从就绪态转变为运行态 ->进入CPU 调度频率高 以毫秒为单位 短期调度也称为进程调度/低级调度 几乎是每个操作系统都有的，一般情况下使用最多

短期调度分为抢占式和非抢占式

# 长期调度

长期调度也称为作业调度/高级调度

在批处理系统中，进程倍放到缓冲池中，长期调度在该缓冲池种选择进程。 分时系统和实时系统中通常不需要长期调度 长期调度控制多到程序设计的颗粒度

# 中期调度

主要用在分时处理系统 中期调度也称为交换调度--换入换出 调度程序根据一定策略临时把内存中的某个进程换出到外存（磁盘缓冲区）或者把外存中的进程换入内存的过程。 可以降低多道程序设计的颗粒度

# 进程的创建

父进程 子进程 父进程创建子进程

    pid=fork();//fork创建了一个进程

    if(pid==0)
    {//pid为0
        printf("this is child process,pid is %d\n",getpid());//getpid返回的是当前进程的PID
        count+=2;
        // execlp("/bin/ls","ls",NULL);// 将ls的功能替换给子进程
        //  当进程调用exec系列函数中的任意一个时，该进程代码段、数据段内容将完全用新程序替代。

    }
    else
    {
        fprintf(stderr,"ERROR:fork() failed!\n");
    }
    printf("count=%d\n",count);
    return 0;
}

# 通信模式：

共享内存：建立起一块供协作进程共享的内存区域，进程通过向此共享区域读或写数据来交换信息。 速度快，但需要一块共享内存区域 共享内存区域在创建他的这个进程的地址空间 共享内存要求两个或者以上的进程之间可以在共享内存内通过读写交换数据。 数据格式与位置由进程决定。

如何实现共享内存： 操作系统实现：如何划分共享内存空间 进程同步实现：具体如何协作 两种共享内存机制： 1.

System V 共享内存
shmget() //建立共享内存
shmat() //建立连接
shmdt() //断开连接

允许多个进程共享同一块物理内存 直接将共享的内存页面通过链接，映射到需要通信的进程各自的虚拟地址空间 进而是得多个进程可以直接访问同一个物理内存页面 1.将共享内存空间映射到各个进程的虚拟地址空间。 2.由操作系统的内存管理单元进行映射。

int mian()
{
    int segment_id;
    char shared_memory;
    const int size=4096;
    // 创建共享内存
    segment_id=shmget(IPC_PRIVATE,size,S_IRUSR | S_IWUSR);
    // 建立链接 返回虚拟内存空间的第一个地址
    shared_memory=(char*) shmat(segment_id,NULL,0);
}

# 第四章 线程

进程是资源分配的单位 进程的上下文切换会带来额外的开销

线程是CPU调度的最小单元

# 线程的概念：

将资源和指令执行分开 一个资源+多个指令执行序列 线程保留了并发的优点，避免了进程切换的代价 实质就是映射表（资源）不变而PC指针（指令执行序列）变

# 多线程模型

# 多线程的优势：

  1. 效率高

  2. 高响应度

  3. 资源共享

  4. 经济 共享资源，代价较小。且线程之间的切换代价小

  5. 多处理器体系结构的利用 能充分使用多处理器的体系结构，是每个线程并行运行在不同的处理器中，不管有多少CPU，单线程只能运行在一个CPU上，在多CPU上使用多线程加强了并发功能。

# 线程的共享和独享

线程共享代码段、数据段、文件、信号、堆 注意：堆是在进程空间中开辟出来的 -> 堆也会被线程共享

线程独享：

每个线程都有自己的寄存器和栈

  1. 线程ID
  2. 寄存器组的值
  3. 线程的栈
  4. 错误返回码
  5. 线程的优先级
  6. 线程的信号屏蔽码
  7. 程序计数器

# 线程库

  • Unix不支持线程 但是可以在用户层面类支持多线程

用户空间的线程库：

  • 在用户空间中提供一个没有内核支持的库。
  • 此库的所有代码和数据结构都存在于用户空间中。
  • 调用库中的一个函数只是导致了用户空间的一个本地函数调用，而不是系统调用

内核空间的线程库：

  • 执行一个由操作系统直接支持的内核级的库。
  • 此时，库的代码和数据结构存在于内核空间中。
  • 调用库中的一个API函数通常会导致对内核的系统调用。

# 用户级线程： 多对一模型

对设置了用户级线程的系统，调度仍以进程为基本单位。 （内核不知道有线程的存在，对内核来说能被他调度的还是进程）

用户级线程的优势：

  1. 线程切换不需要转换到内核空间，节省开销。
  2. 调度算法可以是进程专用的。不同进程根据自身情况选择调度算法。
  3. 用户级线程的实现与操作系统平台无关。

用户级线程的缺点

  1. 系统调用的阻塞问题。
  2. 不能充分利用多处理机多重处理的特点。

# 内核级线程： 一对一模型

在内核中为每一个线程设置了线程的控制块，感知线程的存在并且加以控制。

优势：

  1. 在多处理机系统中，内核能够同时调度一个进程中的多个线程。
  2. 如果进程中的一个线程被阻塞了，内核可以调度该进程中的其他线程。
  3. 内核支持的线程具有很小的数据结构和堆栈，切换代价比进程小。
  4. 内核本身也可以设计多线程技术，提高系统给的执行速度和效率。

缺点： 内核开销大

# 用户级线程+核心极线程： 多对多模型

现在大多数操作系统都是多对多模型

三种主要的线程库：

  1. POSIX Pthread
  • Pthread作为POSIX标准的扩展，可以提供用户级或内核级的库

线程不是和进程并发的，当一个线程被进程创建后，进程会等这个线程运行完。

  1. Win32
  • Win32线程库是适用于Windows操作系统的内核级线程库
  1. Java
  • ava语言都是用Java虚拟机解释和运行的，Java虚拟机本身运行在用户态，如果要支持线程，那只能支持到use-level状态
  • 如果一个操作系统已经支持线程，则Java虚拟机可以调用内核关于线程的函数

线程同步异步处理 线程的取消 如果一个线程调用exec函数，怎会替换整个进程，包括这个线程。

# 线程池

  • 在进程开始的时候会创建一定数量的线程，放到线程池中等待工作。

  • 当服务器收到请求时，它会唤醒池中的一个线程（如果有可以用的线程），并将要处理的请求传递给它。

  • 一旦线程完成了服务，它会返回到池中再等待工作。

  • 如果池中没有可用的线程，那么服务器会一直等待直到有空线程为止。

优点：响应速度快

# 第五章 CPU调度

  • 多道程序设计是并发的

程序代码可以分为 计算类代码 和 I/O类代码

CPU约束型： 以计算为主，CPU区间会较多。 一般在后台静默运行

I/O约束型： 以I/O为主，会有较短的CPU区间配合I/O。 例如QQ、微信等。

等待时间

周转时间 周转时间=等待时间+运行时间

响应时间 响应比=周转时间/执行时间

CPU运行和执行I/O的时间与调度算法无关。

# 调度算法：

1. 先到先服务 一般多用于批处理系统

甘特图 注意画法

等待时间：从进入队列到开始执行 平均等待时间： $(T_p1+T_p2+T_p3)/3$ 每个进程的总等待时间除进程个数

周转时间：从进入队列到执行完毕 平均周转时间： 周转时间=等待时间+进程运行的时间 $(TT_p1+TT_p2+TT_P3)/3$

2. 短作业优先调度

对于每一个任务，按照CPU区间的长度，先执行CPU区间短的，若CPU时间相同，则按照后顺序

优点：保证平均等待时间最短 缺点：

1. 饿死现象：
如果不断的有短作业进入队列，长作业就一直不能被执行。

2. 估计不准：
需要估计进程的CPU区间，准确性较低，在实际应用中一般不能达到真正的短作业优先

3. 没考虑紧迫程度，只考虑时间长度

如果一个新进程具有更短的CPU时间 非抢占式 SJF： 会允许当前运行的进程先完成其CPU区间 抢占式 SRTF： 可以抢占当前运行的进程，先执行CPU区间更短的(最短剩余时间优先)

抢占式的等待时间计算： 该进程最后一次运行前 非该进程运行的时间总和。 例： 计算P2的等待时间： 在P2最后一个时间片之前，不属于P2的时间片总和：

P1+P3+P1+P4   +   P5 = (20-0) + (42-30)=32

3. 优先级调度 抢占式 和 非抢占式 短作业优先调度 就是 优先级调度的一种特例 （以CPU区间为优先级）

每个任务关联一个优先级，调度优先级最高的任务。

I/O bound任务应该获得更大的优先级，使得IO尽量忙，并且和CPU并行工作

缺点：

  1. 饥饿 低优先权的进程可能永远无法执行，如果不断的有优先级高的进程进入队列

解决方案： 老化 随着时间的推进，进程的优先级逐渐被提高

4.适合交互式的调度 轮转法 Round Robin(RR) 每个进程先获得一个小的时间片q ， 时间片过去之后进入下一个进程的执行。 所以每一个进程等待的时间都不会超过 (n-1)*q

若时间片很大，就接近于先来先服务， 若时间片很小，吞吐量变小，周转时间变长

优点：定时有响应，等待时间较短 缺点：上下文切换次数较多，额外开销多

时间片的设定：80%的CPU区间应该小于时间片 时间片10-100ms，切换时间0.1-1ms(1%)

前台进程：和用户交互 后台进程：批处理

一般前台进程使用轮转法保证交互性
后台进程使用先到先服务
前台队列和后台队列一般用优先级调度，前台队列的优先级较高

时间片轮转法不可能导致饥饿现象

存在问题：

  1. 没法区分I/O bound和CPU bound

解决饥饿现象： 多级反馈队列

传统算法： 通常在使用多级队列调度算法时，进程进入系统时被永久地分配到一个队列。例如，如果前台进程和后台进程分别有独立队列，进程并不从一个队列转移到另一个队列，这是因为进程并不改变前台或后台性质。这种设置的优点是低调度开销，缺点是不够灵活。

多级反馈队列中，任务可以在队列之间移动，更细致的区分任务

主要思想: 1. 根据不同CPU区间的特点以区分进程。 2. 如果进程使用过多CPU时间，那么它会被转移到更低优先级队列 3. 在较低优先级队列中等待时间过长的进程被转移到更高优先级队列

总结： 可以根据“享用”CPU时间多少来移动队列，阻止“饥饿” 最通用的调度算法，多数OS都使用该方法或其变形，如UNIX、Windows等。

CPU调度总结：

  1. 并发能提高效率 并发的核心是进程能让出CPU
  2. 进程让出CPU 下一个进程使用CPU 这个选择就是调度
  3. 进程、线程(内核级、用户级)都能调度 任务调度
  4. 调度任务分类:交互式，批处理
  5. CPU调度算法:FCFS,SJF,Priority(批处理);RR(交互式)
  6. CPU调度算法:多级队列,多级反馈队列(混合)

对调度器来说，为什么操作系统要区分I/O-bound和CPU-bound很重要。

  1. IO约束需要的优先级较高，注重和用户的交互
  2. IO型约束 使用的时间片一般比较短， CPU型约束 使用的时间片一般比较长。

# 第六章 进程同步

在每个进程里面都是顺序执行的

多到程序设计中一定会并发。 并发执行： 间断性、失去封闭性、不可再现性

两个进程共享一块共享内存，需要保证进程同步。

并发进程： (进程同步的根源)

在一个进程的工作没有完成之前，另一个进程就可以开始工作，这些进程就称为可同时执行的。或者称它们具有并发性，并且把可同时执行的进程称为并发进程。

并发进程之间的关系： 进程互斥： 多个进程不能同时使用同一个资源，某个进程使用该资源时，其他进程必须等待

进程同步： 多个进程的调用存在时序关系，某些进程的执行必须先于另一些进程 合作进程间有意识的行为，进程之间有一定的依赖关系，有一定的同步机制保证它们的执行次序

典型例子：公共汽车上司机与售票员的合作。只有当售票员关门之后司机才能启动车辆，只有司机停车之后售票员才能开车门。司机和售票员的行动需要一定的协调。

竞争条件：

  1. 多个进程并发访问和操作共享数据
  2. 和时间有关就是和调度顺序有关
  3. 这样的错误非常难调试

避免竞争条件： 一个进程在访问共享内存的时候，其他进程停止访问。

临界区： 临界区是一段代码，每个进程都有一段代码，，进程就是通过这段代码来访问临界区的，这段代码包含了对共享内存的修改、更新等操作，这段代码就叫做临界区。 就是一次只允许一个进程进入的一段代码 。

容易混淆的问题：

  • 进程之间的共享变量都应该放入临界区内吗？ 不是的，只读但是不修改的情况不需要放入临界区。

  • 一个进程只有一个临界区？ 不是

  • 有N个进程共享一个变量，是不是总有一个进程在临界区里。 不是，只需要保证，有进程在临界区时其他进程不要访问临界区。

临界区问题 临界区里是每个进程分别对共享变量的那段代码，怎样确保，当有一个进程i正在自己的临界区执行时，没有任何其它进程j也在它(进程j)的临界区中执行

仅仅通过“一次只允许一个进程进入的一段代码（互斥）”这一个限制，不能严谨的解决问题

临界区问题必须满足三项原则：

  1. 互斥 如果进程Pi在其临界区内执行，那么其他进程都不能在其临界区内执行
  2. 前进 如果没有进程在其临界区内执行，且有进程需进入临界区，那么只有那些不在剩余区内执行(在进入区)的进程可参加选择，以确定谁能下一个进入临界区，且这种选择不能无限推迟。
  3. 有限等待 从一个进程做出进入临界区的请求，直到该请求允许为止，其他进程允许进入其临界区的次数有上限。在不冲突的前提下，多个进程在想访问共享资源时能够访问到

适用于两个进程在临界区于剩余区交替执行 turn 表示轮到谁进临界区 flag 表示哪个进程已经准备好了进入临界区

// procrss0

    // 进入区
    flag[0]=true; // 表示process0已经准备好进入临界区
    turn=1; // 表示轮到process1进入临界区

    '''
    进入临界区
    '''
    // 退出区
    flag[0]=false; // process0已经完成了进入临界区的操作
    // 剩余区

}while(true)

// process1
do{
    flag[1]=true; // 表示process1已经准备好进入临界区
    turn=0; // 表示轮到process0进入临界区
    while(flag[0] && turn==0); // 等待 进程0的临界区时间结束

    '''
    进入临界区
    '''
    flag[1]=false; // process0已经完成了进入临界区的操作
}while(true)

  2. 在Peter算法中，有可能会有进程i反复进入临界区(True),但是 当进程J提出要进入临界区时，操作系统下一次一定会调度进程J(False) 由操作系统的调度算法来决定（内核级调度）

# 3. 信号量算法（操作系统提供的工具）// 核心级的实现

以上 软件算法或者是硬件指令都使用while循环来进行忙等待。 他们都是用户自己设计的算法，都受制于操作系统的调用

信号量算法： 内核级的算法

PV信号量： PV信号量是一种实现互斥与同步的模型，许多系统提供了系统调用来实现P/V功能，但仍需要程序员自己来完成同步

struct semaphore S
{
    intvalue;//记录资源个数
    PCB*queue;//等待在该信号量上的进程
}
P(semaphores);//消费唤醒操作wait()
V(semaphores);//产生唤醒操作signal()

PV操作由P操作和V操作组成，也称P操作原语和V操作原语。 对信号量S的操作只能由P、V操作来进行，即： 信号量S（value、queue）数值的改变只能由P、V操作完成

P(S):将信号量S.value减1，若结果小于0，则把调用P(S)的进程放入该信号量的等待队列

P(semaphore s)
{
    s.value--;// 消耗一个资源
    if(s.value<0) sleep(s.queue);
}

V(S):将信号量S.value加1，若不大于0(小于等于0)，则唤醒一个信号量S等待队列中的进程。

P(semaphore s)
{
    s.value++;// 释放一个资源
    if(s.value<=0) wakeup(s.quequ);
    //从s.queue上取出一个进程;将其放入就绪队列
}

现在，PV操作真的能够让进程B进入“阻塞”状态 一旦发现资源不够用了，就直接将申请该资源的进程B阻塞，不再会调度进程B，直到这个资源被A释放，才将B唤醒 为什么PV操作能达到这样的功效？ 因为PV是系统内核来执行的

# 第七章 死锁

定义：多个进程因循环等待资源而造成无法执行的现象

后果：

进程无法执行 造成资源浪费

造成死锁的资源： 只读文件：只读文件是不会造成进程等待，也就不会死锁 打印机：打印机一次只能让一个进程使用，就会造成死锁

资源互斥访问是死锁的必要条件（即一次只能有一个进程访问的资源）

死锁指一组进程，一个进程不存在死锁

死锁的特征：

  1. 资源是互斥的
  2. 发生了死锁时，每个进程都占有一些资源，并且等待一些资源

# 死锁的四个必要条件：（注意和临界区问题的四个条件区分）

  1. 互斥使用 互斥是资源的固有特性
  2. 不可抢占 资源只能自愿放弃
  3. 占有和等待 进程必须占有资源再去申请
  4. 循环等待 在资源分配图中存在一个环路 有环路不一定会死锁，但是没有环路一定不会出现死锁现象。

# 消除死锁现象

# 1. 死锁预防（不常用）

破坏死锁的四个必要条件之一

方法1.互斥使用 （一般不会破坏这个条件） 方法2.破坏条件占有并等待 不等待：进程在开始执行前请求并分配到它需要的所有资源（实现比较困难） 不占有：只有当进程没有拥有任何资源时才允许它去申请资源 缺点：需要预知未来，编程困难；许多资源分配后很长时间后才使用，资源利用率低，饥饿现象； 方法3:破坏非抢占 1.如果一个进程占有资源并申请另一个不能立即分配的资源，那么其现已分配的资源被隐式地释放。 2.如果一个进程申请一些资源，那么首先检查它们是否可用,如果可用，那么就分配它们;如果不可用，那么检查这些资源是否已经分配给其他等待额外资源地进程。如果是，那么就从等待进程中抢占这些资源，并分配给申请进程 方法4:破坏“循环等待”：对所有资源类型进行完全排序，且要求每个进程按递增或者递减的顺序来申请资源。这样就不可能出现循环等待资源的情况。 缺点:如果编程时就需考虑，用户会觉得很别扭

破除死锁的必要条件会引入不合理因素，实际很少使用

# 2. 死锁避免（比较常用）

检测每一个资源请求，如果造成死锁就拒绝 安全序列： 如果系统中的所有进程存在一个可完成的执行序列P1，...Pn，则称进程处于安全序列 如果系统能按某个顺序为每个进程分配资源（不超过其最大值）并能避免死锁，那么系统状态就是安全的 前一个进程资源释放之后，后一个就可以使用用资源并且执行则称该执行序列为安全序列 死锁避免的实现措施：防止系统进入不安全状态

银行家算法 解决死锁问题 当资源有很多个实例的时候，资源分配图已经不适应了。

allocation 已经分配的资源 Max 最大需要的资源 Available 空闲的资源 Need 进程还需要的资源 Work 执行步骤（最终的安全序列）

对于每一次执行，先遍历所有的进程，如果当前的进程 没有被完成，且当前所空闲的资源 + 已分配给当前进程的资源 >= 该进程需要的最大资源，则执行该资源，标记该进程已完成，已分配给该进程的资源释放到可用资源中，进行下一次循环知道所有进程都被执行。按照执行的顺序就得到了一个安全序列

答题步骤：

  1. 求出 Need 矩阵 即 Max字段值减去Allocation字段值
  2. 一步一步求出当前执行的进程，并且更新avaliable字段
  3. 得出安全序列

Resource-RequestAlgorithm 用于解决 进程执行过程中，又提出更多的资源请求

算法步骤：

  1. 如果新的request资源小于原来的Need（否则不执行）
  2. 如果request 小于 Available
  3. 假装分配 将 request 的值添加到Allocation中 Need 减去request avaliable 减去 request 中的值 判断这样分配是否会造成系统死锁，如果造成死锁就拒绝该次资源请求 都则就分配资源

# 第八章 内存管理

内存是现代计算机运行的中心 程序必须装入内存之后，才能以进程为单位被内存解释、执行

内存使用：将程序放到内存中，PC指向开始地址

重定位： 为了执行程序，对程序的地址做修改 物理地址空间：实际的内存空间（8G、16G） 逻辑地址空间：虚拟的内存空间 内存管理就是将逻辑地址空间映射到物理地址空间

# 如何完成从逻辑地址空间到物理地址空间的映射？

需要特定的硬件设备：内存管理单元MMU

# 1.确定进程可访问的合法地址的范围，并确保进程只访问其合法地址

基地址寄存器 界限地址寄存器

用基地址+界限地址（偏移量）

# 什么时候进行重定位？

编译、载入、执行

# 1. 在连接的时候进行重定位

编译时：效率高，但是程序只能放在内存固定的位置上 例如：卫星、飞机上的嵌入式系统

# 2. 可重定位代码

载入时：慢一些但灵活，但是一旦载入内存就不能移动。

# 3. 运行时重定位（最合适的时机）

每执行一条指令，都要做一次地址翻译（逻辑地址->物理地址）

步骤：

  1. 先从PCB中取出基地址
  2. 做重定位 通过软件来实现太慢了，一般使用硬件来实现

程序执行的具体步骤： 操作系统在内存中找到一块空闲内存 将该空闲内存的初始地址赋给该进程的PCB 将该程序加载到空闲内存中 执行中，PCB的基址也就是CPU基址寄存器 每取指执行一条指令，都要进行地址翻译 程序开始执行起来

# 交换

覆盖技术和交换技术都是扩充内存的重要方式 覆盖技术：一个进程不需要把所有的指令和数据都装入内存，而是将程序划分为若干个功能相对独立的程序段，按照程序的逻辑结构让那些不会同时执行的程序段共享同一块内存区。这样，使用户感觉到内存扩大了，从而达到内存扩充的目的（主要在同一个作业或者同一个进程内执行）

**交换技术：**先将内存某部分的程序或数据写入外存交换区，然后再从外存交换区中调出指定的程序或数据到内存中让其执行的一种内存扩充技术。

# 多重分区分配

动态存储分配问题：

  1. 首次适应 ：分配第一个足够大的，能够满足当前进程需求的孔
  2. 最佳适应 ：分配最小的足够大的孔 （最有可能形成小的外部碎片）
  3. 最差适应 ：分配最大的孔

# 碎片

外部碎片： 内存块加起来能满足一个请求，但是由于不连续（中间有断层），不能用来连续分配 降低外部碎片的有效方式：紧缩

挪出一块完整的空闲区

紧缩的代价高： 在紧缩过程中，CPU将内存中的数据复制到另一个区域，无法完成其它任务 紧缩会改变进程的地址

内部碎片： 分出去的分区略大于请求的内存长度。这个余下的小内存块属于该分区，但是无法利用

# 分段

支持用户视角的内存管理

程序是有许多段代码、数据组成的

进程的大小：reqSize=code+data+stack+heap code和data不难处理，stack和heap难处理

code和data是不变的 stack和heap是动态增长的

为stack和heap预留空间 如果预留空间用完了之后就找一个更大的空间，移动该进程

各个段区别对待，分别分配

# 将逻辑的、用户视角的段映射成实际的物理内存空间

编号+段表映射

分段的优点：

  1. 不同的段有不同的含义，可区别对待
  2. 每个段独立编址，编程容易(如果是一个大的一维地址空间，程序员一会儿就糊涂了!)(分治)

分段的缺点：

  1. 空间预留；空闲空间很大却不能分配；内存紧缩
  2. 著名的碎片概念:空闲的却用不上的空间

# 分页

  1. 将程序的逻辑地址空间分成若干页
  2. 将物理地址的空间分成若干块
  3. 建立逻辑地址的页和物理地址的块的映射表（页表）

根据逻辑地址去查页表，找到物理地址 -- 通过内存管理单元来实现

先判断逻辑地址和物理地址的位数 逻辑地址:页号+偏移量 物理地址:帧号+偏移量

相关公式：

平均每个页都会浪费1/2个帧 解决方法： 帧小一些 --> 页面小 --> 页号多 -->页表大 适量的小一些

页表的访问 页表保存在内存中 访问一次有效数据需要访问两次内存

有效访问时间 (内存存取周期+快表存取周期) * 快表命中率+(2 * 内存存取周期 +快表存取周期) * (1-快表命中率);

快表：本质是寄存器，支持并行搜索

# 页表结构

1.基寄存器 2.长度寄存器

分页的本质就是时间换空间

给页表分页：

  1. 层次页表 核心：将页表再分页，将页表的逻辑地址拆分为多张页表 顶层页表常驻内存，不需要映射的逻辑地址不需要建立页表项

分为 外部页表 和 内部页表

  1. 哈希页表 主要思想：
    1. 虚拟地址（逻辑地址）的页号经过哈希函数转换后，指向页表中某个页表项
    2. 哈希函数值相同的虚拟页号，指向同一个页表项，它们在那个页表项下组成一个链表
    3. 地址翻译时，由虚拟页号哈希后锁定对应链表，搜索链表得到虚拟页号的匹配项
    4. 如果找到匹配项，则找到了虚拟页号对应的物理页帧

哈希页表的缺点是: 1. 在同样的哈希表上映射多个页面引起的冲突。 2. 如果多个页表项映射在同个入口处，则可能导致其相应的哈希入口负担过重，从而在链表上搜索的代价非常高。

当一个程序占用大的虚拟地址空间的一小部分时，哈希页表非常合适。

  1. 反向页表 不给每个进程建表，而是给物理内存建表

任何一个物理帧都可以在反向页表中找到对应的页表项 对于整个操作系统而言，有且只有一个反向页表

优点：存储空间比较小 缺点：有可能需要查找整个表来寻求匹配，增加了查找页表所需要的时间

分段技术符合人的习惯，但是空间低效。

现代操作系统使用的内存分配方式：段页结合

用户产生逻辑地址（分段），从内存角度上将每一段分页

查询原理 逻辑地址包括 段号+偏移量， 段号->查段表->得到基地址 基地址+偏移量->页号 页号->查页表->物理页号

# 第九章 虚拟内存

分段、分页是操作系统管理内存的两种主要方式 虚拟内存是实现分段与分页的关键核心

进程的局部性原理： 时间局部性： 如果程序中的某条指令一旦执行，不久后该指令可能再次执行； 如果某数据被访问过，不久以后该数据可能再次被访问；

空间局部性： 一旦程序访问了某个存储单元，不久后，其附近的存储单元也将被访问，即程序在一段时间内所访问的地址，可能集中在一定的范围以内

通过局部性原理： 在程序装入时：

将程序的一部分装入内存，其余部分留在外存，就可以启动程序执行。 在程序执行过程中，当所访问的信息不在内存时，由操作系统将所需要的部分从外存换入内存，然后继续执行程序。 操作系统将内存中暂时不使用的内容换出到外存上，从而腾出空间存放将要调入内存的信息。

虚拟存储技术： 建立了一个内存外存的两级存储结构。 利用局部性原理实现高速缓存。

请求时才映射

请求、换入、建立映射 ---- 懒惰的交换

虚拟内存的优点：

  1. 地址空间>物理内存
  2. 部分程序放入物理内存

如何实现虚拟内存：

  1. 将需要的部分先放入内存。
  2. 不经常使用的部分，先不放入内存

需要使用不在内存中的部分时（当访问没有映射的虚拟地址时），请求调页（换入）

没有空闲页框时，从当前物理帧里淘汰出去一帧，腾出空间 设计淘汰算法，避免刚换进来就被淘汰出去这种现象

页面淘汰算法（页面置换算法）：

  1. FIFO算法
  2. 最优的淘汰算法：MIN（OPT）算法
  3. LRU算法

FIFO

缺页中断率： 缺页次数/访问次数

belady异常： 给进程分配了更多的物理帧数，但是缺页次数却变多了。

MIN算法: 选最远将使用的页淘汰。是一种最优的方案，可以证明缺页数最小!

LRU页面置换 算法(LeastRecentlyUsed)： 用过去的历史预测将来。 LRU算法:选最近最长一段时间没有使用的页淘汰(最近最少使用)。

实现方法：

  1. 每页维护一个时间戳（代价比较高）
  2. 维护一个页码栈

LRU近似实现-----将时间计数变为是和否（二次机会算法） 循环扫描各个页，这个页被访问时会将该页的引用位置为1，每次扫描到该页时，若该页的引用位为1，则置0，若为0，则淘汰该页。

定时（可控制）清除R位

抖动（颠簸）现象

防止抖动现象的根本手段：给进程分配足够多的帧 进程的工作集：

  1. 在某段时间内，进程要访问的页面的集合
  2. 反映了进程在接下来一段时间内很可能会频繁访问的页面的集合

驻留集：

  1. 实际分配给进程的物理帧数
  2. 为了防止抖动，驻留集的大小要大于工作集

银行家算法

在操作系统的设计中，用户态指非特权状态，内核禁止此状态下的代码进行潜在危险的操作，比如写入系统配置文件、杀掉其他用户的进程、重启系统等。

操作系统可以执行特权指令

中断处理是操作系统必须提供的功能

设置时钟的指令运行在内核模式

# DMA

# 批处理系统和分时操作系统

批处理系统： 不具备交互性 就是指用户将一批作业提交给操作系统后就不再干预，由操作系统控制它们自动运行。

分时操作系统：

分时操作系统是指在一台主机上连接多个带有显示器和键盘的终端，同时允许多个用户通过主机的终端，以交互方式使用计算机，共享主机中的资源。 分时操作系统将CPU的时间划分成若干个片段，称为时间片，操作系统以时间片为单位，轮流为每个终端用户服务

# 并发

在操作系统中，并发是指一个时间段中有几个程序都处于已启动运行到运行完毕之间，且这几个程序都是在同一个处理机上运行，但任一个时刻点上只有一个程序在处理机上运行。 从宏观上看程序是一起执行的，但从微观上看程序还是顺序执行的。

# 并行

在操作系统中是指，一组程序按独立异步的速度执行，无论从微观还是宏观，程序都是一起执行的。
